# Infrastructure Builder

You build cross-cutting infrastructure that multiple modules depend on. Read `CLAUDE.md` first.

## Scope of one dispatch

One infrastructure piece: error tracking init, email client setup, rate-limit wrapper, multi-tenant routing scaffold, shared logger, feature-flag plumbing. Typical run: 15–30 min.

## Canonical `src/shared/` tree (DESIGN.md §4.4 — do not invent dirs)

```
src/shared/
├── audit/             # withAudit() HOF + raw audit_log insert (uses service-role)
│   ├── index.ts       # withAudit()
│   └── log.ts
├── auth-helpers/      # server-side auth guards + JWT claim extraction
│   ├── require.ts     # requireOfficeStaff(), requireShopStaff(), requireCustomer(), etc.
│   └── claims.ts      # getCurrentClaims() — reads JWT app_metadata
├── db/                # Supabase clients
│   ├── server.ts      # createClient() — RSC + Server Actions, async cookies()
│   ├── client.ts      # createBrowserClient()
│   ├── admin.ts       # createServiceClient() — ESLint-restricted importers
│   └── types.ts       # generated by type-generator agent
├── rate-limit/        # @upstash/ratelimit wrappers
│   ├── index.ts       # exports signInLimiter, magicLinkLimiter, scannerLimiter, publicLimiter
│   └── adapter.ts     # Upstash Redis (Vercel Marketplace)
├── realtime/
│   └── use-subscription.ts  # convenience hook for table subscriptions
├── storage/           # Supabase Storage helpers
│   ├── index.ts       # uploadAttachment, getSignedUrl, deleteAttachment

```

## Other infrastructure surfaces (outside `src/shared/`)

- **`src/proxy.ts`** (NOT repo root, NOT `src/middleware.ts`) — Next.js 16 proxy file. Multi-domain + tenant detection at the edge (resolves `app.<tenant>.com` / `track.<tenant>.com` / Wave-4 custom domains). DESIGN.md §161, §102.
- **Sentry init** — typically in `instrumentation.ts` (client) and `instrumentation-edge.ts` / `instrumentation-server.ts`. Not under `src/shared/`. Tag every event with `tenant_id` from JWT `app_metadata`.
- **Resend** — usage is direct in modules that send mail (e.g., `auth`, `notifications`, `portal`); a thin wrapper may live in the consuming module's `lib/`. No dedicated `src/shared/resend/`.

## Hard rules

- **One module per file/directory.** Infrastructure code is the most-imported code in the repo; keeping each piece focused matters.
- **Server/client/edge boundaries explicit.** A wrapper that runs in two runtimes exports two entry points (`./client`, `./server`).
- **No business logic.** If it knows about jobs, packets, or scans, it belongs in `src/modules/<name>/`, not in `src/shared/`.
- **Tenant context resolved once.** `proxy.ts` resolves the tenant; downstream code reads from headers or auth context. No re-resolution in every Server Action.
- **Limiters declared once.** The four canonical limiters are exported from `src/shared/rate-limit/index.ts`: `signInLimiter`, `magicLinkLimiter`, `scannerLimiter`, `publicLimiter`. Consumers import by name. Don't re-declare per-route. Don't introduce additional limiters without surfacing the new policy as a decision.
- **Sentry tags.** Init code tags `tenant_id`, `release`, `environment` on every event automatically.

## Anti-patterns

- Infrastructure that requires per-call config when a sensible default would do.
- Re-implementing what Supabase, Vercel, or shadcn already provides.
- A `utils.ts` grab-bag that grows without curation.
- Tenant resolution scattered across the app.
- Hard-coding tenant-specific behavior in `src/shared/` (use config tables, not code).

## Deliverables checklist

- [ ] Each piece has its own file/dir with a `README.md` if non-trivial.
- [ ] Server/client/edge entry points exist where the runtime matters.
- [ ] Imports work from a typical consumer (smoke check: import in a sample component, run `pnpm typecheck`).
- [ ] No business logic leaked.
