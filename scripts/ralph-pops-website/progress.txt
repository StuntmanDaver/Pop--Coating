It finds `src/proxy.ts` at the monorepo root (the SaaS app's middleware) when building pops-website. - Shipping ZIP locked at 33811 (open question §12.1 unresolved)
- Logo canonical: /images/Pops-no-border.png
- pops-website uses `apps/pops-website/app/` for routes (NOT `src/`) — no src-dir convention here
- pops-website .gitignore must exclude node_modules, .next, *.tsbuildinfo to avoid committing build artifacts
- Tailwind v4 is CSS-first: `@import "tailwindcss"` + `@theme { ... }` in app/globals.css; postcss.config.mjs uses `@tailwindcss/postcss`. NO tailwind.config.ts.
- Design tokens (color-pops-yellow-*, color-ink-*, semantic, font-display/text/mono, radius-*, shadow-1..4) are sourced VERBATIM from docs/design/popsindustrial-design-principles.md §10. Do NOT diverge — those values are paint-chip-locked.
- Tailwind v4 auto-generates utility classes from `@theme` tokens (e.g. `bg-ink-800`, `text-pops-yellow-500`, `font-display`, `rounded-sm`, `shadow-3`) — no need to declare them.
- next/font CSS variables (`--font-display`, `--font-text`) are set on `<html>` via the className spread; they CASCADE-OVERRIDE the static fallback stacks declared in `@theme`. The `font-display` / `font-text` Tailwind utilities resolve `var(--font-*)` at the element, so the next/font self-hosted family wins. Self-hosted assets land at `/_next/static/media/*.woff2` — never the Google Fonts CDN.
- Static assets are in `apps/pops-website/public/` (URL: `/images/<file>`, `/pdfs/<file>`, `/favicon.ico`). Server-side content imports live at `apps/pops-website/content/` (e.g. `content/site-globals.json` for footer / nav columns). NEVER serve content/site-globals.json over HTTP — it's an `import` target only.
- Source of static assets: `.firecrawl/popsindustrial/assets/{images,pdfs}/` — 194 images + 4 PDFs. If a story needs an image, check this directory first; the file is already at `/images/<exact-filename>` after US-004.
- Multi-resolution `favicon.ico` was generated via `magick <png> -define icon:auto-resize=16,32,48,64,128,256 favicon.ico`. Re-run that command if `cropped-favicon.png` is ever updated.
- Components live at `apps/pops-website/components/<area>/<file>.tsx` (kebab-case files, PascalCase named exports). Areas: `layout/`, `marketing/`, `ui/`, `forms/`, `seo/` (per the per-AC notes scattered through US-006..US-049).
- Layout primitives use named exports + `forwardRef` to accept refs and pass through HTML element props (`ComponentPropsWithoutRef<"div">` for Container, `ComponentPropsWithoutRef<"section">` for Section). React 19 still supports forwardRef; use it where ACs explicitly require ref forwarding.
- className merging convention until clsx lands: `className ? \`${base} ${className}\` : base`. shadcn primitives (US-008+) will introduce `cn()` from `lib/utils.ts`; switch to that once available.
- Tailwind v4 ships a default 4px spacing scale, so `py-16` / `md:py-24` (= 64px / 96px) match `--space-16` / `--space-24` from design principles §5.1 without declaring a custom `--spacing-*` namespace in `@theme`.
- Container caps at `max-w-[1280px]` with responsive horizontal padding `px-4 sm:px-6 lg:px-8`. Don't lower the cap; the design contract assumes 1280px on every page (hero, contact form, service grid).
- Eyebrow / micro-label convention (design principles §4.3 + §10): `text-xs` (12px) Inter 600 uppercase with `tracking-[0.04em]`. Default brand tone uses `text-pops-yellow-500`; secondary "ink" tone uses `text-ink-400` — the canonical "secondary text" color (per token comment in §3.2). Uppercase is reserved for micro-labels, button text, and the wordmark — never headings.
- `lib/utils.ts` exports `cn(...inputs)` = `twMerge(clsx(inputs))`. Use it in every component that takes a `className` prop or composes Tailwind class lists. Deps already added: `clsx`, `tailwind-merge`. Replace the `className ? \`${base} ${className}\` : base` template-literal merge in older primitives (Container/Section/EyebrowLabel) when next touched.
- shadcn-style `cva()` from `class-variance-authority` is the convention for components with multiple visual variants (Button, future Card, future Input). Define a single `cvaXxx` recipe with `variants` + `defaultVariants`, expose `VariantProps<typeof recipe>` to props, and merge via `cn(recipe({...}), className)`.
- `asChild` pattern via `@radix-ui/react-slot`: when consumers want the component to project its styles onto a child element (typically `<Link>` or `<a>`), accept `asChild?: boolean` and render `Slot` instead of the native element. Slot forwards props and merges classes onto the immediate child. The native element variant is the default — preserve focus, type attributes, ARIA, etc. on it.
- Focus ring convention: `focus-visible:ring-2 focus-visible:ring-pops-yellow-300 focus-visible:ring-offset-2 focus-visible:ring-offset-ink-800` + `outline-none`. Tailwind v4 implements `ring-*` via `box-shadow`, so the ring color (`pops-yellow-300` = `#FFE067`) appears as a 2px outer halo with a 2px ink-800 offset between it and the element — matches `--shadow-focus` from design principles §10. Use `focus-visible` (not `focus`) to avoid showing the ring on mouse click.
- Loading-state width preservation: render the original children invisibly (`aria-hidden + invisible`) so the layout box keeps its measured width, then absolutely-position the spinner in `absolute inset-0`. This avoids a "snap" when toggling `isLoading` mid-submit. Use `aria-busy={isLoading || undefined}` on the button.
- Internal preview pages live at `apps/pops-website/app/dev/<feature>/page.tsx` (NOT `_dev/` — Next.js App Router treats `_folder` as a private folder that is opted out of routing). The `dev/*` namespace must be excluded from the sitemap (US-042) and `robots.ts`. These are working previews for browser verification of components — keep them around as a reference panel for the design system.
- Tailwind v4 wraps the `hover:*` utility under `@media (hover: hover)`, so hover styles only fire on devices with a fine pointer. In headless Chromium under Playwright that media query evaluates true — so `hover:shadow-3` works in browser-verification — but mobile tablets and touch devices won't see hover states (this is the correct behavior for design principles §6.3 service-tile lift; touch users get the same affordance via tap → navigate, not via persistent hover).
- Card sub-component composition: Card itself owns `bg + border + radius + p-6` so consumers don't double-pad; CardHeader/CardContent/CardFooter are organizational only (margin/flex, never padding). CardTitle defaults to `<h3>` for semantics — wrap in your own heading element via `asChild`-style cloneElement OR just render the right tag inside the card if you need an h2/h4 instead.
- CardDescription uses `opacity-80` (not a separate muted color token) so it adapts to whatever tone the parent Card is on (dark→ink-100 muted, light→ink-800 muted) without a context provider. Override with an explicit `text-*` class via `className` if a darker description is needed.
- Form input primitives (Input/Textarea/Label) use a NATIVE element pattern (no Radix wrappers) because each is a single-element control where the browser default ARIA semantics are already correct. Surface styling lives in one place per file (no cva — only one visual variant). Error state is styled via the dual selector `data-[error=true]:border-danger-500 aria-[invalid=true]:border-danger-500` so consumers can use either: react-hook-form sets `aria-invalid` automatically, while a manually-controlled form can set `data-error`. Both fire the same red-border treatment.
- The form input focus-visible halo uses the SAME ring tokens as Button (US-008): `focus-visible:border-pops-yellow-500 focus-visible:ring-2 focus-visible:ring-pops-yellow-300 focus-visible:ring-offset-2 focus-visible:ring-offset-ink-800`. The border-color change to `pops-yellow-500` matches design principles §6.4 ("2px `--pops-yellow-500` border + `--elev-focus` ring") — the visual reads as a single yellow halo because the inner border and outer ring share the same family.
- Labels use `font-medium` (Inter 500) per AC, intentionally lighter than CardTitle's `font-bold` (700). This is the right hierarchy: a label is a micro-prompt, not a heading. Required-field asterisk pattern is `<Label>{label}<span className="text-pops-yellow-500"> *</span></Label>` — wired up properly in US-011 FormField wrapper, but the asterisk color comes from the design principles §6.4 spec.
- `placeholder:text-ink-400` is the canonical placeholder color on dark inputs. Don't reuse it for labels — placeholders disappear on focus, labels persist; mixing them causes B2B field confusion (per §6.4 "never floating placeholders").
- `resize-y` on Textarea (vertical-only resize) prevents users from breaking the form layout by dragging horizontally. It's the convention for content-bearing fields like message/project-details.
- FormField (US-011) is the wiring contract for accessibility on every form field: it owns the Label/htmlFor pairing, the helper-or-error `<p id={`${name}-help`}>`, and uses `React.cloneElement` to inject `id`, `name`, `aria-invalid`, and `aria-describedby` onto the single input child. Consumers only pass `<Input />` / `<Textarea />` as children — they NEVER set those four attrs themselves. If the consumer already sets `id`/`name`/`aria-describedby`, FormField preserves them (cloneElement uses `child.props.x ?? name` for id/name and joins existing `aria-describedby` tokens with the help id). Error string takes precedence over helperText, and both render with `mt-2 font-text text-sm` — error in `text-danger-500`, helper in `text-ink-400`.
- Use `Children.only(children)` to enforce a single React child at runtime — failing fast with a clear error if a consumer wraps multiple inputs in one FormField. Pair with `isValidElement<P>()` (typed generic) for the cloneElement narrowing — this is the only way to keep TS strict-mode happy without `any`.
- Radix primitive wrappers (Checkbox, future RadioGroup/Switch/Select/Dialog/Dropdown/Tabs) MUST start with `"use client"` — Radix uses React Context + hooks internally and will throw if imported from a Server Component. Apply this rule to every new file under `components/forms/` or `components/ui/` that imports from `@radix-ui/*`.
- For Radix Root + Indicator primitives (Checkbox, RadioGroup, Switch), the canonical "double-element" pattern is: Root carries the interactive semantics + tap target + focus ring; an `aria-hidden` inner element carries the visible chrome that responds to `data-state` via `group-data-[state=*]`. This decouples the visual size (e.g. 18x18) from the WCAG 2.1 AA tap-target size (44x44 per design principles §9). Use `inline-flex items-center justify-center` on a `h-11 w-11` Root with a smaller centered child — never use `padding` to grow the target, since that would shift the visual box.
- Use `forceMount` on Radix's `Indicator` subcomponents to keep them in the DOM at all times so `opacity` / `scale` transitions can fire on state change. Without `forceMount`, the indicator unmounts on state flip and re-mounts on the next state — transitions never get a starting frame to animate from.
- Scraped URLs in `content/site-globals.json` are absolute (`https://popsindustrial.com/...`). When rendering links inside the rebuild, run them through a `toRelative(url)` helper that uses `new URL(url).pathname` and strips the trailing slash — never pass absolute live-site URLs into `<Link>`. Apply this rule to footer nav, header (US-014), and any future story that consumes scraped href data.
- Footer-style nav lists give every link a `min-h-11` (44px) tap target via vertical padding (`py-2`) so touch users get a WCAG-compliant target without enlarging the visible text. Pair with `space-y-3` on the `<ul>` to keep adjacent targets from overlapping. Text-only links inside content (vs nav lists) don't need this.
- Use `inline-block` (not the default inline) on a link that needs `min-h-*` to take effect — height/min-height are ignored on inline elements. The footer nav links use `inline-block min-h-11 py-2` for this reason.
- JSON imports (`import data from "../../content/site-globals.json"`) work in both Server and Client Components because tsconfig has `resolveJsonModule: true`. The import is fully typed by TS literal inference; cast to a richer interface (e.g. `Record<string, ColumnEntry[]>`) when you need a discriminated-union shape. The build inlines the JSON — don't ship `content/*.json` to `public/` or it'll be served twice.
- For mixed-shape lists where some entries are `{label, href}` and others are `{label, value}` (the Visiting Us footer column has hour `value` rows, not links), use a discriminated union with a type-narrowing predicate (`isLinkEntry`). Render each shape as a different element — never render `<a href="">{value}</a>` as a fallback because it would create a useless empty link.
- `useId()` from React works in Server Components — use it for ANY id that is referenced by an aria-* attribute in the same component (e.g. `aria-labelledby`, `aria-describedby`, `htmlFor`/`id` pairs that are NOT already wired by FormField). This makes the component safe to render multiple times on a single page without id collisions. Do NOT add `"use client"` just to use `useId()`.
- `<Button asChild variant="..."><Link href="...">label</Link></Button>` is the canonical Button-styled router link pattern. Used by Hero (US-015) and any future marketing surface that needs a CTA pointing to an internal route. Preserves Button's variant colors, focus ring, hover state, and tap target while next/link delivers prefetch + RSC navigation. Don't reach for `<Link className="<button-classes>">` — that diverges whenever Button styles change.
- `next/image` with `fill` (used for hero background photos) requires the parent to have `position: relative` (or absolute/fixed). Pair `relative isolate` on the parent so the overlay div + content div live in the same stacking context — the overlay is `absolute inset-0` (no z-index), and content sits on top via `relative z-10`.
- Decorative background images on hero/marketing surfaces use `alt=""` (empty string, NOT null). Screen readers skip empty-alt images entirely; the heading + lede already convey meaning. Reserve descriptive alt text for images that ARE the message (leadership headshots, before/after job photos, certifications).
- Display heading sizes are not yet declared as Tailwind tokens (the @theme block in app/globals.css does not include `--text-display-*`). Until a future story adds them, marketing components hit the §3.3 spec via arbitrary values: `text-[36px] md:text-[56px]` for display-lg with `leading-[1.05]`. Same approach applies for display-xl (72px), display-md (44px), display-sm (30px) when other components need them.
- Tailwind v4's `scale-[1.03]` compiles to the modern CSS `scale: 1.03` property, NOT `transform: scale(1.03)`. So `getComputedStyle(el).transform` will read `none` even when the scale is applied — verify hover scale via `getComputedStyle(el).scale` instead. Both properties compose with `transform-origin` and respect `transition-transform` (which v4 transitions: transform, translate, scale, rotate together — see the transition shorthand emitted in `.transition-transform`).
- Shop photography for service tiles lives at `/images/<service>-*.jpg` — pick the largest (non-thumbnail) variant available; next/image handles responsive crops. Avoid the `-300x*` and `-200x*` thumbnails as `src` (they degrade on retina). Examples used in US-016: `industrial-painting-lakeland-fl-1024x683-1.jpg`, `abrasive-media-blasting.jpg` (no -300x suffix), `industrial-powder-coating-lakeland-fl-IMG_3687.jpg`, `large-capacity-powder-coating.jpg`. The `.firecrawl/popsindustrial/manifest.json#alt_text_audit` flags which images need bespoke alt text — the rebuild closes that gap as it consumes them.
- Card with edge-to-edge image: override Card's default `p-6` with `p-0` via className, wrap the image in `<div class="relative aspect-[16/10] overflow-hidden">`, and put padded content (eyebrow/title/body) in a sibling `<div class="p-6">`. Keep `overflow-hidden` on the Card too so the image clips into the rounded corners. The `interactive` Card variant (`hover:shadow-3`) still fires when an inner Link is hovered — `:hover` bubbles to ancestors.
- For interactive tiles where the WHOLE card navigates: Card outside, Link inside, content inside Link. Put the focus ring on the Link (`focus-visible:ring-2 ring-pops-yellow-300 ring-offset-2 ring-offset-ink-800`) and rely on `:hover` bubbling for the Card's `hover:shadow-3` lift. Add `group` to the Link so descendants can opt into `group-hover:*` (e.g. image scale). Don't use `role="link"` on the Card — it makes the Link redundant for assistive tech.
