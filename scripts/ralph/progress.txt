## 2026-05-01 — US-016 (pgTAP RLS test suite — all 31 tests green)
- **What was implemented:** Three root causes fixed across two prior iterations:
  1. jwt_helpers.sql: SECURITY DEFINER added to set_jwt_for_staff/customer/workstation (RLS blocked SELECT after SET ROLE authenticated with no JWT)
  2. migration 0011: REVOKE table-level UPDATE on jobs FROM authenticated + GRANT back specific columns (column-level REVOKE was no-op with table-level grant present)
  3. migration 0012_app_schema_usage_grant.sql: GRANT USAGE ON SCHEMA app TO authenticated, anon (app schema had NULL nspacl — authenticated couldn't call app.* functions directly even with EXECUTE grants)
  4. jwt_helpers.sql: plan(1)/pass()/finish() wrapper added (valid TAP file required — no_plan without tests raises "No tests run!" error from pgTAP _finish())
  5. test_function_authorization.sql: plan(6)→plan(7), throws_ok→lives_ok for stale-version test (claim_workstation returns ok:false, no exception), nested-$$ syntax removed
- **Files changed:** supabase/migrations/0012_app_schema_usage_grant.sql (created), supabase/tests/helpers/jwt_helpers.sql, supabase/tests/rls/test_function_authorization.sql, scripts/ralph/prd.json.
- **Learnings for future iterations:**
  - Postgres schema USAGE is distinct from function EXECUTE. Even with EXECUTE granted, authenticated cannot reference app.* without GRANT USAGE ON SCHEMA app.
  - RLS policy expressions work without schema USAGE because they're evaluated under the table owner's privileges — only explicit client-side calls need USAGE.
  - pgTAP's finish() raises "No tests run!" if called with no_plan() and zero assertions. Use plan(1) + pass() for pure setup files.
  - throws_ok with NULL error code is for "any exception". Use lives_ok when no exception is expected.
  - Nested $$ dollar-quoting inside throws_ok causes syntax errors. Simplify the SQL string.
